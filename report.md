# Document Data Engineering Case Study — Report

## Executive summary

A five-stage Databricks pipeline — ingest, preprocess, chunk, index, classify — runs end-to-end on Free Edition serverless compute in 23m 34s, processing 2,484 unique CVs from the Kaggle resume dataset. PySpark handles ingestion and preprocessing per the brief; classification uses Llama 3.3 70B via the Foundation Model API with a JSON-schema `responseFormat` constraining output to the five required brackets plus a confidence score. The headline finding is a heavily senior-skewed corpus — 76% of resumes land in 10+, with model confidence on that label averaging 0.95 against 0.77–0.83 elsewhere. The headline challenge was building a production-pattern pipeline — medallion tables, Vector Search, Foundation Model API, and Asset Bundles — within Databricks Free Edition's constraint set: one Vector Search endpoint, no Direct Vector Access (forcing Delta Sync), pay-per-token Foundation Models only, and serverless-only compute. A supporting blocker was external storage access — the auto-created Access Connector in Free Edition's locked managed resource group is unusable due to deny assignments, so a second was created in an unlocked resource group and registered via CLI rather than SQL.

## Methodology

The pipeline is structured into five stages — ingest, preprocess, chunk, index, classify — wired as tasks in a single Databricks Job. `ingest` → `preprocess` runs serially; `chunk` and `classify` then fan out from `preprocess` and run in parallel, with `index` sequenced after `chunk`. Each stage is a separate Python entry point in one wheel artefact, deployed via Databricks Asset Bundles. Stage outputs follow medallion-architecture naming: `cv_bronze` (raw bytes plus a content hash), `cv_silver` (extracted text), `cv_silver_chunks` (per-chunk text + 1024-dim embedding), `cv_gold` (silver plus experience bracket and LLM confidence). A Mosaic AI Vector Search index is layered on top of `cv_silver_chunks` to satisfy the brief's "prepared for embedding/retrieval/analysis" requirement.

The Kaggle resume dataset sits in Azure Blob storage, surfaced to Databricks through a Unity Catalog external location. Ingestion uses Spark's `binaryFile` reader for recursive PDF discovery and `sha2(content, 256)` for content-based deduplication; 2,484 unique PDFs survived. The Kaggle layout groups CVs into one folder per profession, so the parent-directory name is parsed out of `path` at ingest and stored as a lowercase `category` column, propagating through silver and gold. Preprocessing uses a `pypdf`-backed Python UDF for text extraction with `regexp_replace` whitespace normalisation. The chunk stage applies LangChain's `RecursiveCharacterTextSplitter` (1,600-char windows, 200-char overlap, paragraph→newline→sentence→space separators), then embeds every chunk via `ai_query('databricks-gte-large-en', …)`, storing the 1024-dim vector in a Delta column. The index stage creates (or reuses) a Vector Search endpoint and a Delta Sync Index over `cv_silver_chunks.embedding`. Classification calls `ai_query('databricks-meta-llama-3-3-70b-instruct', …)` with a JSON-schema `responseFormat` whose `bracket` field is enum-constrained to the five required values, guaranteeing a legal label; a sibling `confidence` field acts as a soft signal for thin or contradictory resumes.

A `--limit` flag was added to the classify stage and surfaced as a job-level parameter, enabling cheap smoke tests (50 rows) before committing to the full pay-per-token run.

### Assumptions

- **All CVs are in English.** The Kaggle dataset is English-only, and the chosen embedding model (`databricks-gte-large-en`) is monolingual. A multilingual production setting would need `bge-m3` or a multilingual GTE variant.
- **`pypdf` strips most layout.** Output is a single text string per CV with paragraph breaks largely intact but visual structure (columns, tables, sidebars) collapsed. Section headers (`EXPERIENCE`, `EDUCATION`, `SKILLS`) typically survive in capitals because they're emitted as caps in the source. Section-aware parsing was considered and de-scoped — heuristics on capitalised headers are brittle across the heterogeneous layouts in the corpus, and the recursive splitter already respects natural paragraph boundaries without needing the structure.

### Constraints

- **PySpark for ingestion and preprocessing.** Mandated by the brief for scalability. At 2,484 rows the architecture is over-engineered relative to a single-node pandas pipeline; the trade is deliberate — the same code scales to 100k+ documents without rewriting.
- **Databricks Free Edition.** One Vector Search endpoint × one unit, Delta Sync indexes only (no Direct Vector Access), Foundation Model API pay-per-token only (no provisioned throughput, no GPU), serverless compute only.
- **LLM as add-on analytics, not production.** Per the brief, classification is an analytical pass rather than a productionised service: no retry-with-backoff on transient `ai_query` failures, no per-row cost monitoring, no shadow-mode comparison against a prior model — all of which a production deployment would add.

## Architecture decisions

- **Five tasks, not one.** Failure isolation, separate task logs, and free Delta caching between tasks outweigh the modest orchestration overhead. Re-running classify after a prompt change costs nothing upstream; re-running chunk after a parameter change leaves the rest of the pipeline alone.
- **Bundles, not DLT.** DLT's declarative model fits the LLM step poorly, and the dataset is small enough that streaming or incremental ingest add no value. Bundles also keep local pytest-driven development against `databricks-connect` straightforward.
- **`ai_query` over an external provider.** No API key or secret scope, Spark-native parallelism, and native structured-output support; the equivalent OpenAI/Anthropic call would require either single-threaded driver calls or a hand-rolled threadpool wrapped in a Python UDF.
- **Llama 3.3 70B over 405B.** Equivalent quality on short-context classification at lower cost and latency.
- **`databricks-gte-large-en` for embeddings.** Alibaba GTE-large-en-v1.5 (434M params, 1024-dim, 8192-token context, 65.39 MTEB) — state-of-the-art within its size class for English retrieval. Hosted by Databricks behind the same `ai_query()` mechanism as classification, so no second auth surface. The 8192-token window comfortably fits 1,600-char chunks (~400 tokens).
- **Self-managed embeddings + Delta Sync Index.** Direct Vector Access is unsupported on Free Edition, so Delta Sync is the only legal index. Embeddings are written to a Delta column rather than computed inside the index: vectors stay SQL-inspectable, the embedding step is just another `ai_query()` call (uniform medallion story), chunking can be tuned without re-embedding inside the index, and the vectors are portable if the index is ever rebuilt.
- **Chunk size 1,600 chars / 200 overlap.** Resumes are short, dense documents — sub-256-token chunks fragment specific role claims; 1,000+-token chunks blur the centroid. 1,600 chars (~400 tokens) is the sweet spot. The 200-char (~12.5%) overlap stops single-sentence facts ("5 years at Goldman as a quant") being split between two chunks where neither carries the full claim. Recursive separator order (`\n\n` → `\n` → `. ` → ` ` → `""`) respects natural paragraph and sentence boundaries before falling back to whitespace.

## Key findings

The 2,484 resumes distribute heavily toward seniority: **76% land in 10+**, 14% in 7-10, 7% in 3-5, ~1.5% in 5-7, and ~1% in 0-2. Average self-rated model confidence on 10+ is **0.95**, against 0.77–0.83 elsewhere — when the model commits to 10+ it does so confidently, and when it picks elsewhere it is markedly less sure, supporting a genuinely senior-leaning corpus over a hedging model. Self-rated confidence is a soft signal, not a calibrated probability, so this is suggestive rather than definitive.

Only one row fell below confidence 0.6 — a CONSTRUCTION resume of 17.5 KB with confidence exactly 0. (An earlier 6k-char run had two such outliers; the second resolved to a confident classification once more text was visible.) The residual case suggests **0-2 acts as a fallback bucket** for resumes whose date signal cannot be extracted — likely a structurally awkward PDF rather than a thin career history.

The non-monotonic dip at 5-7 (41 rows, the smallest bracket despite a wider population window than 0-2) survived the cap extension, supporting a **bracket-boundary effect**: when torn between "around 5" and "around 7", the model appears to hop to a neighbouring bracket rather than commit to the middle.

End-to-end wall time for the full pipeline on a serverless cluster was **23m 34s**: ingest 1m 14s, preprocess 6m 46s, chunk 1m 58s, index 21s, classify 15m 32s. `chunk` and `classify` fan out after `preprocess` and run in parallel, so `classify` (the LLM calls) drives the post-fork branch on its own; the `pypdf` UDF in `preprocess` is the second dominant cost. Ingest, chunk, and index combined account for under 3m 30s.

The retrieval surface was validated via a contrastive-query test from a workspace notebook. The in-domain query "python data engineer with spark and aws experience" returned a top-5 band of `[0.00225, 0.00246]` with relevant engineering chunks (Spark/AWS terms, data-engineering role descriptions). A deliberately out-of-domain control ("unicorns in a tree?") returned a *different* top-5 band of `[0.00171, 0.00174]` with a different set of chunks — predominantly "Additional Information" / hobbies sections. Vector Search uses distance-style scores rather than [0, 1] cosine, so absolute values are tight; the in-domain vs out-of-domain delta — not the absolute number — is the signal that the embedding + ANN surface is doing real semantic work.

Profession is captured at ingest by parsing the parent-directory name out of `path` and propagated through silver and gold as a `category` column. The Kaggle layout yields **24 categories**, mostly flat at ~100–120 rows each with three smaller folders (`bpo`, `automobile`, `agriculture`).

## Challenges

- **Truncation cap.** Resume text is truncated to 10,000 chars before classification to bound input cost. An initial 6k cap was raised after the first full run revealed **969 of 2,484 resumes (39%) exceeded 6k**; 10k closes 86% of that gap. Because truncation keeps the *first* N chars, the earliest role — establishing career start date — is most likely to be cut, biasing toward underestimating experience. The before/after run moved **38 rows from 7-10 into 10+** (76.3% vs 74.9%): exactly the expected direction, small enough to confirm the dominant skew is a corpus property rather than a cap artefact. The 26 mega-resumes over 15,000 chars are the residual concern; head-plus-tail concatenation would close that gap without expanding the context window.
- **UDF observability.** A first end-to-end run showed `silver.write` at ~7 min and a follow-up `silver.filter(...).count()` at another ~6 min — the lazy DataFrame still carried the `pypdf` UDF in its query plan, so post-write counts silently re-ran the entire UDF. Re-binding to `spark.table("cv_silver")` for any post-write reads cut preprocess wall time from 13m 21s to 7m 25s. The class of bug pure-Python unit tests cannot surface — only a cluster run with a task timeline reveals it.
- **Bracket ambiguity.** "Years of experience" is undefined in the brief: total working years, years in a specific field, or relevant years? The prompt was pinned to count paid roles and internships and exclude education and personal projects, but a production recruiter use case would need this nailed down per role and probably split into multiple labels.

## What would extend this with more time

- Manual labelling of a 50-resume sample to measure LLM accuracy against ground truth — and to interrogate whether the 10+ skew survives review.
- Head-plus-tail prompt construction for the residual mega-resumes (>15,000 chars), where even the 10,000-char head can miss the earliest start date.
- Hybrid retrieval (vector + BM25) and reranking via the Databricks-hosted `bge-reranker` for higher precision on top-k.
- Section-aware chunking once the corpus is more uniform — splitting on `EXPERIENCE` / `EDUCATION` / `SKILLS` boundaries would let recruiters filter retrieval to specific resume sections.
- Cost projection at production scale: Foundation Model API per-token pricing applied to expected monthly CV volume, alongside a budget for retries on `__EXTRACTION_ERROR__` rows handled by an OCR fallback.
